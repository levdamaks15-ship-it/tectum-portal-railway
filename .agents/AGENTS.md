### Git Push
- **Запрет автоматического пуша:** Никогда не выполнять и не предлагать `git push` без явного подтверждения от пользователя.

- Миграции столбцов в `lifespan()` должны проверять тип драйвера (`postgresql` vs `sqlite`) и выполняться только для соответствующей БД.

## 3. Номенклатура продуктов

- **«Шифер 8 волн рифленый»** — строго утверждённое название. Всегда использовать именно его.

## 4. Данные и роли

### Полная свобода данных (единый кабинет)
- Все данные абсолютно общие. Любой мастер смены (роль `master`) имеет право просматривать любые графики, сводки, выработку и расходы сырья других мастеров, а также свободно вносить и редактировать любые рапорты смен.
- Администраторы (`admin`), технологи (`technologist`), директора (`director`) имеют полный доступ ко всем данным.
- **Правила изоляции мастеров НЕ применяются.** Исторические данные не фильтруются по `master_id`.

### Единый рапорт мастера
- Авторизация: мастер заходит под единой ролью «Мастер смены» (PIN или SSO), затем выбирает конкретное имя из выпадающего списка.
- **Отсутствие концепции открытия/закрытия смен для пользователей.** Жизненный цикл active/closed скрыт. Баннер текущей смены не отображается. Мастер заполняет независимые сменные рапорты за выбранную дату/смену/линию.
- **Единая форма ввода:** Производство (листы, сбросы, замесы), переборка (на склад ГП, 1 сорт), брак Дестакера (11 полей), расход/приход сырья — всё в одной форме.
- Исключение брака СКК/ОТК: поле брака СКК скрыто с формы мастера (учитывается в отдельной системе).
- Масса выработки рассчитывается на клиенте в кг и тоннах с точным совпадением названий продуктов.
- Терминология: «Разборщик» → «Дестакер».

### CRUD и администрирование

**Разрешено редактирование для сущностей:**
- `models.Master` — сотрудники
- `models.ProductNorm` — нормативы расхода
- `models.Downtime` — простои
- `models.DowntimeDirectory` — справочник простоев
- `models.MonthlyPlanBoard` — план-факт доска

**План-факт доска (`MonthlyPlanBoard`):**
- Обновляется через Upsert (`POST /api/plan_board`).
- Все изменения (CREATE, UPDATE, DELETE) логируются в `AuditLog` с подробным описанием старых и новых значений.

**Администраторы (`admin`):**
- Имеют права на полное редактирование (UPDATE) и удаление (DELETE) смен, отчётов ЛФМ, партий и простоев (даже если смена закрыта в БД).
- Все изменения логируются в `AuditLog` со сравнением старых и новых значений.

**Безопасность DOM на фронтенде:**
- При изменении стилей/классов/атрибутов всегда проверять существование элемента: `const el = document.getElementById(id); if (el) { ... }`
- Категорически запрещено вызывать `.style` или `.classList` напрямую от `document.getElementById`.

## 5. Защита целостности базы данных (Database Integrity)

**Общий принцип:** Любое изменение кода не должно нарушать целостность данных в БД. При редактировании кода всегда учитывать существующую структуру данных.

### 5.1. Проверка внешних ключей (Foreign Keys)
- **Перед созданием/обновлением записи**, которая ссылается на другую таблицу (например, `shift.master_id` → `masters.id`, `monthly_plan_board.master_id` → `masters.id`), **всегда проверять**, что целевая запись существует.
- Если запись не найдена — возвращать ошибку 400/404, а не создавать битую ссылку.
- **Запрещено** записывать `master_id`, `shift_id` или любой другой внешний ключ без валидации его существования в родительской таблице.

### 5.2. Транзакционная безопасность
- Групповые операции (создание смены + отчёта ЛФМ + партии) должны выполняться в одной транзакции.
- При ошибке в любой части — полный `rollback()`, чтобы не осталось «полу-созданных» записей.
- Все `db.commit()` должны предваряться проверками, а при ошибке — `db.rollback()`.

### 5.3. Защита эндпоинтов от 500-ошибок
- Все `GET` эндпоинты, возвращающие списки (`/api/shifts/all`, `/api/plan_board`, `/api/report/summary` и т.д.), **обязательно** должны иметь try/except блок.
- При сериализации связанных моделей (`.master.name`, `.master.id`) **всегда** проверять, что связанный объект не `None`.
- Если связанная запись отсутствует — выводить плейсхолдер («Н/Д», `null`, `""` или `0`), а не падать с `AttributeError`.

### 5.4. Защита от удаления связанных данных
- **Запрещено удалять мастера**, если на него ссылаются смены (`shifts`) или план-факт доска (`monthly_plan_board`). Код должен проверять наличие связанных записей перед удалением (уже реализовано в `delete_master`).
- При удалении смены — каскадно удалять связанные отчёты ЛФМ, партии и простои (уже реализовано в `admin_delete_shift`).

### 5.5. Graceful degradation (отказоустойчивость)
- Если данные в БД повреждены (например, битый `master_id`, неверный формат даты), эндпоинт должен **продолжить работу** и вернуть частичные данные, а не падать с 500.
- Использовать конструкцию:
  ```python
  master_name = s.master.name if s.master else "Мастер удалён"
  ```
  для всех связанных полей.

### 5.6. Миграции без потери данных
- Добавление новых колонок — только через `ALTER TABLE ADD COLUMN` с `try/except` (уже реализовано в `lifespan()`).
- Переименование/удаление колонок — только если точно известно, что на них нет зависимостей в коде.
- **Запрещено** менять тип существующей колонки, если в ней уже есть данные (может вызвать потерю данных).

### 5.7. Бэкапы перед крупными изменениями
- Перед выполнением массовых операций (очистка данных, импорт из Excel, изменение схемы БД) агент должен предложить пользователю сделать бэкап базы данных.
- Если это продакшн-БД (PostgreSQL на Railway) — предложить сделать дамп через `pg_dump` или через встроенные бэкапы Railway.

### 5.8. Миграции исторических данных (Historical Data Cleanups)
- Изменение логики синхронизации или обработки данных в API (например, устранение багов, приводящих к появлению "фантомных" записей) **не исправляет** уже поврежденные исторические данные в базе.
- При исправлении багов синхронизации всегда проверяйте необходимость исправления старых данных. **Для продакшена на Railway (где ручной запуск скриптов затруднен)** предпочтительным способом является добавление одноразовой логики исправления напрямую в функцию **`lifespan()`** (FastAPI startup event). Это гарантирует автоматическое применение Data-миграции сразу после деплоя. Административные эндпоинты (как `/api/admin/...`) следует использовать только для тех исправлений, которые требуют ручного контроля или ввода параметров пользователем. Не оставляйте "мусор" в БД после исправления бэкенд-логики.

### 5.9. Нечеткий поиск (Fuzzy Matching)
- Названия производственных линий могут храниться в БД и приходить с фронтенда в разных форматах (например: `ЛФМ-1`, `Линия 1`, `Линия 1 (ЛФМ-1)`).
- При написании запросов в SQLAlchemy избегайте строгого равенства `==` для текстовых идентификаторов линий. Используйте подстрочный поиск (`models.Shift.line.like("%1%")` или `"1" in value`), чтобы избежать потерь данных при синхронизации и объединении отчетов.

## 6. Производственный план (План-факт)

### Фиксированные нормы (не расчёт от месячного плана)
- Обычный день (план): **2700 листов**
- Ночь (план): **3300 листов**
- Санитарный день (понедельник, смена «День»): **0 листов**

### Факт (fact_sheets)
- Формируется на основе `LFMReport.lfm_sheets` (общее количество сформованных листов от машиниста ЛФМ).
- Любые изменения отчётов ЛФМ (создание, изменение, удаление) автоматически обновляют `fact_sheets` в `MonthlyPlanBoard` через `sync_lfm_to_plan_board()`.

### Sanitary Day (понедельник, смена «День»)
- План = 0.
- Если фактическое производство есть — оно полностью записывается (перевыполнение плана).

### Сетка отчётов (Reporting Architecture)
- План статичен. Отчёты (графики, недельные таблицы) всегда генерируют полную сетку ожидаемых смен (например, 14 смен за неделю) со статическими плановыми числами.
- Фактические данные накладываются поверх этой сетки.

## 7. Расчёт производительности и потерь

### Цикл формовки
- Один цикл = **26 секунд**.
- За цикл выпускается **2 листа**.
- Исключение: для «Шифер 7 волн 3500*980» — **1 лист** за цикл.

### Потери при простое
- Расчёт на бэкенде динамически:

  $$\text{Потери (тонн)} = \frac{\text{Длительность (сек)}}{26} \times \text{Выход листов за цикл} \times \frac{\text{Вес листа из ProductNorm (кг)}}{1000}$$

## 8. Журнал простоев

- **На портале** — только форма ввода новых простоев. Историческая таблица простоев за текущую смену удалена. Просмотр/анализ — в Google Sheets.
- **Ленивое создание смен (Lazy Creation):** При просмотре дат/смен бэкенд НЕ создаёт пустые смены (возвращает 404). Смена создаётся на лету только при отправке записи простоя (`create_if_not_exists=true`).
- **Регистрация без медиафайлов:** При регистрации простоев используется только текстовая аналитика. Загрузка фото/видео и запись голоса исключены из формы ввода.
- **Автоэкспорт в Google Sheets:** При создании, изменении или удалении простоя автоматически запускается фоновая задача `sync_downtimes_bg()` → `export_downtimes_to_google_sheets()`.

## 9. Google Sheets интеграция

Вся производственная аналитика дублируется в единый Google Документ (`GOOGLE_SPREADSHEET_ID`).

### Подключение
- Аутентификация: сервисный аккаунт Google через `google_credentials.json` (локально) или переменную окружения `GOOGLE_CREDENTIALS_JSON` (Railway/Render).
- Функция `get_sheets_service()` сначала пробует `GOOGLE_CREDENTIALS_JSON`, затем локальный файл.

### Листы Google Таблицы

| Лист | Функция экспорта | Содержание |
|------|-----------------|------------|
| **Сводный отчет** | `sync_report_to_google_sheets()` | Полная сводка по всем сменам: дата, партия, линия, смена, мастер, продукт, формовка, кондиция, брак, расход сырья, отклонения в %. Вызывается при фоновой синхронизации после сохранения рапорта. |
| **Приход сырья** | `export_receipt_to_google_sheets()` | Данные прихода сырья по сменам: дата, смена, линия, мастер, все виды сырья (хризотил, цемент по силосам, целлюлоза и т.д.). Вызывается при сохранении рапорта и при добавлении/удалении записи прихода. |
| **Нормативы** | `export_norms_to_google_sheets()` | Таблица нормативов расхода сырья по продуктам. Двусторонняя синхронизация: также есть `sync_norms_from_google_sheets()` для обратного импорта. |
| **Справочник простоев** | `export_downtime_directory_to_google_sheets()` | Каталог типовых поломок по участкам/узлам. Двусторонняя синхронизация: также есть `sync_downtime_directory_from_google_sheets()`. |
| **Простои** | `export_downtimes_to_google_sheets()` | Журнал простоев (вызов из main.py через `sync_downtimes_bg`). |

### Правила синхронизации
- **При создании/изменении/удалении простоя** → фоновая синхронизация простоев в Google Sheets.
- **При сохранении/обновлении рапорта смены** → фоновая синхронизация сводного отчёта и прихода сырья.
- **При добавлении/удалении записи прихода сырья** → фоновая синхронизация прихода.
- Синхронизация нормативов и справочника простоев — по запросу из админ-панели (кнопки синхронизации).
- Если `GOOGLE_SPREADSHEET_ID` не задан или равен `1_mock_...` — синхронизация пропускается с логом в консоль.

## 10. Автоматический экспорт в SharePoint

- При сохранении/обновлении рапорта смены (`POST/PUT /api/report`) запускается фоновая генерация Excel-отчёта и загрузка в SharePoint (папка `Reports`).
- **Обработка блокировки (HTTP 423):** Если файл заблокирован пользователем — сохранить локальную копию, залогировать предупреждение, поставить фоновую задачу повторной загрузки (`upload_sharepoint_report_retry`).

### Excel Tables
- Экспортируемые Excel-данные должны быть оформлены как форматированные таблицы (ListObject) с полосатой разметкой и фильтрацией.

## 11. Безопасный импорт данных (Safe Parsing)

- При парсинге Excel-файлов все числовые поля приводить через защитные функции-конвертеры (`safe_int`, `safe_float`) для предотвращения `ValueError` от пробелов, пустых строк, символов форматирования.

## 12. Microsoft 365 интеграция

- Всегда проактивно анализировать возможность интеграции с M365 (SharePoint, Excel Online, Teams, Outlook, MS Graph API).
- Использовать существующую авторизацию MSAL и Graph API.
- Сохранять отказоустойчивость: локальный кэш на случай сбоев сети.
- При разработке интеграций, требующих токенов/ключей, обязательно напоминать пользователю добавить переменные окружения в панель Railway/Render.

## 13. Кэширование (Cache-Busting)

- Все GET-запросы к `/api/*` должны содержать уникальный параметр `?_ts=...` для борьбы с агрессивным мобильным кэшированием.
- При изменении `style.css`, `app.js`, `admin.js` — инкрементировать версию `?v=` в `index.html`/`admin.html`.

## 14. Фильтрация пустых смен в отчётах

- При формировании сводных таблиц (`get_report_summary`) и выгрузке в Google Sheets исключать абсолютно пустые смены, у которых одновременно:
  - `plan_sheets = 0`
  - `lfm_sheets = 0`
  - `warehouse_gp = 0`
  - `zo_batches = 0`
  - не отправлен отчёт по ЗО (`zo_submitted = False`)
  - нет записей простоя (`downtimes`)
*(Примечание: наличие только записей прихода (receipts) не делает смену "не пустой". Такие смены также исключаются из сводных таблиц для предотвращения пустых строк)*

## 15. Автовыбор смены в журналах

- При загрузке журналов (например, простоев) автоматически выбирать последнюю доступную смену и сразу запрашивать данные.

## 16. Принцип отладки

- **Last Known Good State:** При возникновении ошибки сначала найти и проанализировать последнее рабочее состояние (коммиты, бэкапы БД, исходный код). Установить причину, затем вносить правки. Не менять «наугад».

## 17. Переменные окружения

При разработке или изменении интеграций с внешними API (Google Sheets, SharePoint, Telegram Bot и др.), требующих токенов/ключей/идентификаторов (`GOOGLE_SPREADSHEET_ID`, `GOOGLE_CREDENTIALS_JSON`, `M365_CLIENT_ID`, `M365_TENANT_ID`, `M365_CLIENT_SECRET` и т.п.), агент ОБЯЗАН напомнить пользователю добавить эти переменные в панель управления хостинга (Railway/Render). Не предполагать, что локальный `.env` автоматически синхронизируется с облаком.
