## Proactive Slash Command Suggestions

When interacting with the user, ALWAYS proactively suggest the use of slash commands if the context fits. Do not wait for the user to ask about them. 

- **`/goal`**: If the user requests a large, multi-step refactoring, bulk data processing, or a complex setup, explicitly suggest using `/goal` so you can complete it autonomously.
- **`/grill-me`**: If the user is unsure about a design decision or architecture, explicitly suggest using `/grill-me` to have an interactive interview to figure it out.
- **`/learn`**: If you and the user successfully solve a tricky edge-case, custom logic, or setup issue, explicitly suggest using `/learn` to persist this behavior as a skill/rule for the future.
- **`/schedule`**: If a task requires waiting, polling, or running on a timer, suggest using `/schedule`.

## Production Plan Rules
When making changes to the monthly production plan logic or generating plans (e.g. `distribute_plan.py`), ALWAYS use the fixed factory standard norms: Normal day = 2700, Night = 3300, Sanitary day (Monday Day) = 0. Do not calculate proportionally from a monthly target.

**План-факт и данные ЛФМ**: Фактическое количество листов (`fact_sheets`) в План-факт доске (`MonthlyPlanBoard`) формируется на основе общего количества сформованных листов из отчетов ЛФМ (`LFMReport.lfm_sheets`), которые передает машинист ЛФМ. Эти данные должны быть связаны в базе данных: любые изменения (создание, изменение, удаление отчетов ЛФМ) должны автоматически приводить к обновлению соответствующего `fact_sheets` в `MonthlyPlanBoard` для данной смены.

**Прямой учет на Стакере**: Оператор Стакера (`role="stacker"`) регистрирует объем сырого наката напрямую в листах/штуках (поле `stacked_stacks` в базе данных) без использования каких-либо коэффициентов-множителей. На фронтенде и в админ-панели поле ввода называется «Общее количество (листов)» / «Общее количество (Стакер)». В сводной таблице смены отображается один столбец «Стакер (листов)».

**Reporting Architecture**: The plan is static and primary. Reports (daily graphs, weekly tables) MUST always generate the full grid of expected shifts (e.g., 14 shifts for a week) with the static plan numbers, regardless of whether factual data exists for those days. Fact data should be overlaid onto this static plan grid.

**Sanitary Day Facts**: While the plan for a Sanitary Day (Monday Day) is 0, if actual production occurs, it MUST be fully recorded as fact, resulting in an over-fulfillment against the 0 plan.

## Portal Data Architecture and Edit Rules
При работе с сущностями портала Tectum строго придерживайтесь следующих правил редактирования и аудита:
- **Разрешено редактирование (CRUD)** для сущностей: Пользователи/Мастера (`models.Master`), Нормативы расхода (`models.ProductNorm`), Простои (`models.Downtime`). Все изменения должны проходить через соответствующие `PUT` методы.
- **Регистрация простоев без медиафайлов**: При регистрации и ведении простоев используется только текстовая аналитика. Загрузка фото/видео файлов и запись голосовых сообщений (микрофон) исключены из формы ввода. Исторические медиассылки сохраняются в таблицах для обеспечения аудита.
- **План-факт доска (MonthlyPlanBoard)** обновляется через механизм Upsert (метод `POST /api/plan_board`). Все изменения (CREATE, UPDATE, DELETE) в ней должны обязательно логироваться в таблицу `AuditLog` с подробным описанием старых и новых значений.
- **Редактирование закрытых смен и производственных данных**: Администраторы системы (пользователи с ролью `admin`, включая `admin@TectumEngineering.onmicrosoft.com`) имеют права на полное редактирование (UPDATE) и удаление (DELETE) смен, отчетов ЛФМ, партий и простоев (даже если смена закрыта). Все такие изменения должны обязательно логироваться в таблицу `AuditLog` с детальным сравнением старых и новых значений полей.
- **Полная редактура администратором**: Всё, что добавляется на портал, а также все существующие сущности и данные (включая смены, отчеты ЛФМ, партии, простои, склад, сырье) должны иметь возможность полной редактуры (изменение, удаление, принудительное открытие/закрытие смены) администратором системы через интерфейс админ-панели (`/admin`).
- **Безопасные манипуляции с DOM на фронтенде**: При изменении стилей, классов или атрибутов элементов (например, при скрытии/показе блоков или логике аккордеонов) обязательно выполнять проверку на существование элемента (например, `const el = document.getElementById(id); if (el) { ... }`). Категорически запрещено вызывать свойства `.style` или `.classList` напрямую от `document.getElementById` без такой проверки, так как отсутствие элемента вызовет JS-исключение `TypeError` и заблокирует работу всего скрипта.
- **Изоляция исторических данных мастеров**: При формировании исторических и аналитических отчетов на бэкенде (дашборд, аналитика простоев, еженедельная/ежемесячная выработка, экспорт Excel):
  - Если пользователь авторизован как мастер (`user_role == 'master'`), все фактические производственные показатели (выработка, брак, сортность, расход сырья, простои) должны фильтроваться по его ID (`Shift.master_id == user_id`).
  - При этом сетка планов смен (плановые листы/тонны) должна генерироваться полностью в соответствии с правилом `Reporting Architecture` (для сохранения структуры сетки), но фактические данные смен других мастеров должны зануляться (`0`) или скрываться («Смена другого мастера»).
  - Администраторы, технологи и директора должны иметь доступ ко всем данным по всем мастерам.

## Railway Deployment Rules
- **Сбой верификации Python в Mise (Nixpacks):** При развертывании Python-приложений на Railway сборщик `mise` может выдать ошибку `No GitHub artifact attestations found for python`. Для предотвращения этого в корне репозитория обязательно должен находиться файл `mise.toml` со следующим содержимым:
  ```toml
  [settings]
  python.github_attestations = false
  ```
- **Запуск через Procfile:** Для указания команды запуска веб-сервиса на Railway использовать файл `Procfile` в корне проекта:
  ```procfile
  web: uvicorn main:app --host 0.0.0.0 --port $PORT
  ```

## Единый рапорт мастера (Unified Shift Data Logic)

1. **Единый кабинет и авторизация**:
   - Авторизация операторов скрыта на фронтенде. В систему заходит мастер смены в единый кабинет (роль `master`).
   - На экране входа используется единая роль «Мастер смены» с общим PIN-кодом или SSO.
   - Внутри рапорта мастер выбирает конкретное имя из выпадающего списка.
   - **Отсутствие активных смен:** Ручной жизненный цикл открытия/закрытия смен мастерами полностью упразднен. Баннер текущей смены (`#active-shift-banner`) скрывается. Каждый отчет смены сохраняется как независимая запись напрямую через форму.
   - **Полная свобода (без личных кабинетов):** Все данные абсолютно общие. Любой мастер смены (роль `master`) имеет право просматривать любые графики, сводки, выработку и расходы сырья других мастеров, а также закрывать и свободно редактировать любые рапорты смен без ограничений на владение (`shift.master_id != user_id`).
2. **Единая форма ввода**:
   - Мастер вводит все данные за смену в одной форме: производство (Сформовано листов, сбросы наката, количество замесов), переборку (на склад ГП, 1 сорт), брак Дестакера по 11 видам дефектов, расход и приход сырья.
   - Итоговый брак рассчитывается автоматически на клиенте как сумма 11 полей дефектов.
   - Масса выработки рассчитывается на клиенте автоматически в **килограммах** и **тоннах** (выводится внизу блока «Выпуск») с использованием строгого точного совпадения названий продуктов (во избежание багов частичного совпадения подстрок).
   - **Исключение брака СКК / ОТК:** Поле общего брака СКК / ОТК (`#rep-qcd-defect`) скрыто с формы мастера, так как этот брак учитывается контроллерами СКК в отдельной системе.
   - **Терминология:** Роль/устройство «Разборщик» переименовано в «Дестакер». Все подписи брака («Итого брака Дестакера», «Детализация брака Дестакера») приведены к этому виду.
3. **Обработка API**:
   - Единый эндпоинт `POST /api/report` транзакционно создает/обновляет смену (`Shift`), связанный отчет ЛФМ (`LFMReport`), партию (`Batch`) со статусом `"qcd_checked"`, и лог аудита.
   - Допускается редактирование данных закрытых смен администратором с записью подробных изменений в `AuditLog`.
   - Если активная смена открыта Мастером А, то Мастер Б не может вносить или изменять этот рапорт (бэкенд возвращает 403 Forbidden).

- **Безопасный импорт данных из Excel (Safe Parsing)**: При парсинге любых внешних Excel-файлов (таких как рапорты АЦИ или планы выработки) все числовые поля должны приводиться к типам данных с помощью защитных функций-конвертеров (например, `safe_int` и `safe_float`). Это предотвращает падение бэкенда (`ValueError`) при наличии в ячейках пробелов, пустых строк `""`, символов форматирования или некорректных типов данных.
- **Обработка блокировок файлов SharePoint (HTTP 423 Locked)**: При загрузке сгенерированных файлов в облако SharePoint необходимо обязательно перехватывать исключения, связанные с блокировкой файлов пользователями (ошибка `423 Client Error: Locked` возникает, когда файл открыт в Excel Online или на компьютере). Логика программы должна корректно обрабатывать это событие: сохранять локальную резервную копию отчета, логировать предупреждение и информировать пользователя, не прерывая выполнение основных транзакций (например, закрытие смены).
- **Борьба с агрессивным мобильным кэшированием (Cache-Busting)**:
  - Мобильные браузеры (особенно Safari на iOS) склонны агрессивно кэшировать AJAX-ответы (например, статус активной смены), игнорируя заголовки кэша.
  - Для предотвращения возврата устаревших данных все исходящие `GET` запросы к API `/api/*` должны использовать авто-добавление уникального параметра-таймстампа (`?_ts=...`) через глобальный fetch-перехватчик.
  - При любом изменении стилей (`style.css`) или скриптов (`app.js`, `admin.js`) необходимо обязательно инкрементировать версию подключения (параметр `?v=...`) в `index.html` и `admin.html`, чтобы принудительно обновить кэш у клиентов.
- **Синтаксическая целостность и изоляция кода (JS/CSS)**:
  - При изменении CSS-файлов строго контролируйте парность фигурных скобок `{}`. Пропуск одной закрывающей скобки в глобальном правиле ломает валидацию всего последующего CSS, включая медиа-запросы `@media`, что приводит к аварийному отключению мобильной верстки.
  - При редактировании JS избегайте случайного слияния или вложенности функций (например, попадания одной функции внутрь тела другой при заменах). Каждая функция должна быть изолированной, иметь чёткие границы и содержать начальные защитные проверки на существование объектов (`if (!currentUser) return;`).

3. **Бесшовная интеграция:** Использовать существующую авторизацию, Graph API и клиентские библиотеки MSAL для интеграции данных с Office 365, сохраняя при этом отказоустойчивость приложения (например, локальный кэш/хранилище на случай сбоев в сети).


## Автоматический экспорт и автовыбор в интерфейсах

- **Автоматический экспорт в SharePoint**: При любом сохранении или обновлении рапорта смены (через `POST /api/report` или `PUT /api/report/{shift_id}`) бэкенд должен автоматически запускать фоновую задачу по генерации актуального Excel-отчета и его загрузке в SharePoint/Excel Online.
- **Форматированные таблицы (Excel Tables)**: Экспортируемые в Excel данные (например, `excel_exporter.py`) должны обязательно преобразовываться в объект Formatted Table (ListObject) с полосатой разметкой (row stripes) и встроенной фильтрацией/сортировкой.
- **Автовыбор смены в журналах**: Все журналы и списки (например, простоев) при первоначальной загрузке должны автоматически выбирать последнюю доступную смену и сразу запрашивать данные, предотвращая показ заглушки «Загрузка...».
