When interacting with the user, ALWAYS proactively suggest the use of slash commands if the context fits. Do not wait for the user to ask about them. 

- **`/goal`**: If the user requests a large, multi-step refactoring, bulk data processing, or a complex setup, explicitly suggest using `/goal` so you can complete it autonomously.
- **`/grill-me`**: If the user is unsure about a design decision or architecture, explicitly suggest using `/grill-me` to have an interactive interview to figure it out.
- **`/learn`**: If you and the user successfully solve a tricky edge-case, custom logic, or setup issue, explicitly suggest using `/learn` to persist this behavior as a skill/rule for the future.

## Production Plan Rules
When making changes to the monthly production plan logic or generating plans (e.g. `distribute_plan.py`), ALWAYS use the fixed factory standard norms: Normal day = 2700, Night = 3300, Sanitary day (Monday Day) = 0. Do not calculate proportionally from a monthly target.

**План-факт и данные ЛФМ**: Фактическое количество листов (`fact_sheets`) в План-факт доске (`MonthlyPlanBoard`) формируется на основе общего количества сформованных листов из отчетов ЛФМ (`LFMReport.lfm_sheets`), которые передает машинист ЛФМ. Эти данные должны быть связаны в базе данных: любые изменения (создание, изменение, удаление отчетов ЛФМ) должны автоматически приводить к обновлению соответствующего `fact_sheets` в `MonthlyPlanBoard` для данной смены.

**Прямой учет на Стакере**: Оператор Стакера (`role="stacker"`) регистрирует объем сырого наката напрямую в листах/штуках (поле `stacked_stacks` в базе данных) без использования каких-либо коэффициентов-множителей. На фронтенде и в админ-панели поле ввода называется «Общее количество (листов)» / «Общее количество (Стакер)». В сводной таблице смены отображается один столбец «Стакер (листов)».

**Reporting Architecture**: The plan is static and primary. Reports (daily graphs, weekly tables) MUST always generate the full grid of expected shifts (e.g., 14 shifts for a week) with the static plan numbers, regardless of whether factual data exists for those days. Fact data should be overlaid onto this static plan grid.

**Sanitary Day Facts**: While the plan for a Sanitary Day (Monday Day) is 0, if actual production occurs, it MUST be fully recorded as fact, resulting in an over-fulfillment against the 0 plan.

- **План-факт доска (MonthlyPlanBoard)** обновляется через механизм Upsert (метод `POST /api/plan_board`). Все изменения (CREATE, UPDATE, DELETE) в ней должны обязательно логироваться в таблицу `AuditLog` с подробным описанием старых и новых значений.
- **Редактирование закрытых смен и производственных данных**: Администраторы системы (пользователи с ролью `admin`, включая `admin@TectumEngineering.onmicrosoft.com`) имеют права на полное редактирование (UPDATE) и удаление (DELETE) смен, отчетов ЛФМ, партий и простоев (даже если смена закрыта). Все такие изменения должны обязательно логироваться в таблицу `AuditLog` с детальным сравнением старых и новых значений полей.
- **Полная редактура администратором**: Всё, что добавляется на портал, а также все существующие сущности и данные (включая смены, отчеты ЛФМ, партии, простои, склад, сырье) должны иметь возможность полной редактуры (изменение, удаление, принудительное открытие/закрытие смены) администратором системы через интерфейс админ-панели (`/admin`).
- **Безопасные манипуляции с DOM на фронтенде**: При изменении стилей, классов или атрибутов элементов (например, при скрытии/показе блоков или логике аккордеонов) обязательно выполнять проверку на существование элемента (например, `const el = document.getElementById(id); if (el) { ... }`). Категорически запрещено вызывать свойства `.style` или `.classList` напрямую от `document.getElementById` без такой проверки, так как отсутствие элемента вызовет JS-исключение `TypeError` и заблокирует работу всего скрипта.
- **Изоляция исторических данных мастеров**: При формировании исторических и аналитических отчетов на бэкенде (дашборд, аналитика простоев, еженедельная/ежемесячная выработка, экспорт Excel):
  - Если пользователь авторизован как мастер (`user_role == 'master'`), все фактические производственные показатели (выработка, брак, сортность, расход сырья, простои) должны фильтроваться по его ID (`Shift.master_id == user_id`).
  - При этом сетка планов смен (плановые листы/тонны) должна генерироваться полностью в соответствии с правилом `Reporting Architecture` (для сохранения структуры сетки), но фактические данные смен других мастеров должны зануляться (`0`) или скрываться («Смена другого мастера»).
  - Администраторы, технологи и директора должны иметь доступ ко всем данным по всем мастерам.

## Field Migration and Role Data Pattern (Паттерн переноса и миграции полей)
При переносе формы ввода данных или полей между ролями (например, перенос сливов из ЗО к машинисту ЛФМ):
1. **Обратная совместимость БД**: Не удаляйте старые колонки, если они используются в исторических данных или скриптах импорта (например, `import_aci.py`). Добавляйте новые колонки.
2. **Миграция БД**: 
   - Добавьте ALTER TABLE команды для SQLite в `lifespan` функцию в `main.py` с обработкой исключений (`try/except`).
   - Добавьте ALTER TABLE команды для PostgreSQL в `scratch/sync_pg_schema.py`.
3. **Бэкенд**:
   - Обновите Pydantic-схемы (как в `schemas.py`, так и инлайновые в `main.py`), исключив старые поля и добавив новые в соответствующие структуры.
   - Добавьте специализированный эндпоинт (например, `POST /api/shifts/{shift_id}/lfm_drains`) для сохранения перенесенных полей отдельной кнопкой.
   - Обновите расчеты (например, `get_materials_report` для сырья) с учетом новых полей.
4. **Фронтенд**:
   - Обновите соответствующие HTML формы в `index.html` и логику `loadData()` / `submit` функций в `static/js/app.js`.
   - Обязательно добавьте новые поля в форму редактирования смены в админ-панели (`admin.html` и `admin.js`).
5. **Автотесты**: Обновите или создайте автоматический тест-скрипт (например, `scratch/test_material_flow.py`) для проверки интеграции (авторизация, отправка данных, расчеты, сохранение в БД) и запустите его локально перед коммитом.

## Shift Roles and Data Input Logic (Логика ввода данных ролями смены)

При разработке экранных форм и обработчиков API строго соблюдайте разделение обязанностей и логику полей:

1. **Заготовительное отделение (ЗО)** (`role="zo"`):
   - **Вводимые данные**: Расход хризотила (кг по маркам 4-20, 5-65, 6-40), расход цемента (кг по силосам 1, 2, 3, 4), целлюлоза (кг), дробленый шифер (кг), асбозурит (кг), стекловолокно (кг), лапрол (кг), асбокартон (кг) и общее количество замесов за смену (шт).
   - **Логика блокировки**: Форма отправляется один раз за смену, после чего блокируется для оператора ЗО. Редактирование доступно только Мастеру или Администратору.

2. **Листоформовочная машина (ЛФМ)** (`role="lfm"`):
   - **Вводимые данные**: Марка выпускаемой продукции, количество сформованных листов (`lfm_sheets`) и сбросов наката (`lfm_wind_resets`). Сливы смеси за смену (асбест и цемент в кг) также регистрируются на ЛФМ.
   - **Связь с План-фактом**: Суммарный объем сформованных листов с ЛФМ является первоисточником для расчета фактической выработки смены (`fact_sheets`) в `MonthlyPlanBoard`.

3. **Стакер (Укладчик)** (`role="stacker"`):
   - **Вводимые данные**: Номер создаваемой партии, вид продукции, количество сырого наката напрямую в листах/штуках (поле `stacked_stacks` в базе данных) без использования коэффициентов-множителей стоп.
   - **Логика и учет**: Стакер просто укладывает листы сырого наката для передачи в печь на сушку без привязки к множителям или формирования пачек. Подсчет листов Стакера производится напрямую. При создании партии статус инициализируется как `"stacked"`.

4. **Дестакер (Разборщик)** (`role="destacker"`):
   - **Вводимые данные**: Выбор партии со статусом `"stacked"`, кондиция (листов), 1 сорт (листов) и детализированный брак по 11 видам дефектов (скол, сдир, плохой рез, налип снизу/сверху, сломан, упал коробки, вмятина, не соотв. толщины, расслоение, кромка).
   - **Смена статуса**: Успешное сохранение переводит статус партии в `"destacked"`. Сумма брака (`ds_defect`) рассчитывается автоматически на сервере.

5. **Служба контроля качества (СКК / ОТК)** (`role="qcd"`):
   - **Вводимые данные**: Выбор партии со статусом `"stacked"` или `"destacked"`, итоговая кондиция (листов), 1 сорт (листов), итоговый брак (листов).
   - **Сверка**: Успешное сохранение переводит статус партии в `"qcd_checked"`. Если итоговые данные СКК (брак или кондиция) расходятся с данными Дестакера, вся строка этой партии в сводной таблице подсвечивается красным цветом.

6. **Мастер смены** (`role="master"`):
    - **Управление сменностью**: Мастер отвечает за открытие и закрытие смены. Форма открытия смены в контейнере `master-view` (выбор даты, смены и линии) и кнопка «Начать смену» должны оставаться активными и разблокированными на фронтенде, даже если в системе отсутствует текущая активная смена (в то время как формы ввода для других рабочих ролей блокируются).
   - **Изоляция данных разных мастеров**: Если активная смена открыта Мастером А, то для Мастера Б все формы ввода рецептов на складе и открытия смены блокируются на фронтенде (с выводом статуса "СМЕНА ДРУГОГО МАСТЕРА" и блокировкой инпутов), а на бэкенде в методах `create_shift` и `update_receipt` возвращается ошибка `403 Forbidden`. Кнопка закрытия смены доступна только открывшему её мастеру или администратору.
   - **Оформление динамических селекторов партий**: При динамическом заполнении селекторов партий на фронтенде (например, для Дестакера или СКК) в начало списка обязательно добавляется пустая опция по умолчанию `"-- Выберите партию --"` со значением `""`. Это исключает автовыбор первой попавшейся зависшей партии и требует от оператора осознанного выбора перед сохранением.

### Дополнительные правила безопасности, API и информирования
- **Обработка ответов API на фронтенде**: При отправке любых POST/PUT/DELETE запросов из JavaScript (`static/js/app.js`) обязательно должна выполняться проверка статуса ответа сервера (`if (res.ok)`). Запрещается выводить пользователю сообщения об успешном сохранении (например, `alert("Успешно")`) без проверки ответа бэкенда. В случае ошибки (`!res.ok`) пользователю должно быть выведено детальное сообщение об ошибке, полученное из JSON бэкенда (`err.detail`).
- **Строгая авторизация на бэкенде**: Любые POST/PUT/DELETE эндпоинты в `main.py`, вносящие изменения в данные смен, рецептов, простоев или партий, обязаны явно требовать объект `request: Request` и проверять наличие `user_id` и `user_role` в сессии. Если сессия пуста, должна немедленно выбрасываться ошибка `401 Unauthorized` (а не пропускаться проверки ролей по умолчанию).
- **Скрытие простоев для операторов**: У всех операторов (ЛФМ, Стакер, Дестакер, СКК, ЗО) полностью скрыта кнопка/вкладка журнала простоев (`btnDown`). Учет простоев доступен только мастеру, механику, админу, директору и технологу. Закрытие простоев с комментариями по-прежнему закреплено за механиком (`mechanic`).
- **Глобальная панель статусов**: Панель статуса отчетов операторов смены должна располагаться в верхней части страницы (под баннером активной смены) и быть видимой для всех авторизованных ролей. Она должна в реальном времени отображать статус заполнения отчетов (ЗО, ЛФМ, Стакер, Разборщик, СКК) на основе данных текущей смены, помогая операторам координировать ввод данных.
- **Безопасный импорт данных из Excel (Safe Parsing)**: При парсинге любых внешних Excel-файлов (таких как рапорты АЦИ или планы выработки) все числовые поля должны приводиться к типам данных с помощью защитных функций-конвертеров (например, `safe_int` и `safe_float`). Это предотвращает падение бэкенда (`ValueError`) при наличии в ячейках пробелов, пустых строк `""`, символов форматирования или некорректных типов данных.
- **Обработка блокировок файлов SharePoint (HTTP 423 Locked)**: При загрузке сгенерированных файлов в облако SharePoint необходимо обязательно перехватывать исключения, связанные с блокировкой файлов пользователями (ошибка `423 Client Error: Locked` возникает, когда файл открыт в Excel Online или на компьютере). Логика программы должна корректно обрабатывать это событие: сохранять локальную резервную копию отчета, логировать предупреждение и информировать пользователя, не прерывая выполнение основных транзакций (например, закрытие смены).
